# ADR-0002: Представления данных и соглашения между блоками (без привязки к конкретной модуляции)

Дата: 2026-04-21

## Контекст

Тракт связи — это последовательность блоков, которые преобразуют данные на разных стадиях: сообщение, биты, символы, сигнал, воздействие канала, обратные преобразования.

Если не зафиксировать базовые соглашения (что такое “бит”, “символ”, “семпл”, какие dtype/форматы допустимы и где хранить мета-информацию), то код быстро превращается в набор несовместимых блоков и «хаос» в интерфейсах.

При этом цель — **не плодить много классов**, а сохранить простой notebook-first API.

## Решение

### 1) Минимальный набор представлений (data representations)

Мы фиксируем, что блоки обмениваются *простыми типами Python и NumPy-массивами*, без обязательного введения отдельных классов-обёрток.

Рекомендуемые представления по стадиям:

1) **Сообщение (message)**

- `str` — человекочитаемый ввод/вывод (удобно для ноутбука);
- `bytes` — двоичное представление (удобно для “передачи по каналу”).

2) **Биты (bits)**

- `numpy.ndarray` целочисленного типа (например `np.uint8`), значения **строго 0/1**.

Соглашения:

- порядок бит в байте: по умолчанию **MSB-first** (старший бит первым), если не указано иначе;
- форма массива: 1-D (N,), где N — число бит.

3) **Символы (symbols)**

Символ — элемент дискретной последовательности до формирования “временного сигнала” (например элементы созвездия/алфавита).

- `numpy.ndarray` (dtype зависит от схемы):
  - `int`/`float`/`complex` — допускаются;
  - форма 1-D (K,), где K — число символов.

Важное правило: **1 элемент массива = 1 символ**.

4) **Сигнал дискретного времени (samples / waveform)**

Семплы — временная последовательность после “формирования сигнала”.

- baseband (I/Q): `numpy.ndarray` комплексного типа (`np.complex64/128`)
- passband (реальный): `numpy.ndarray` вещественного типа (`np.float32/64`)

Правило: **1 элемент массива = 1 семпл**.

### 2) Метаданные (sample rate, carrier, etc.)

Мета-информация не должна “теряться” в голых массивах.

Решение:

- Общие параметры прогона и сигналов храним в `ctx` (контексте), который передаётся через pipeline:
  - `sample_rate` (Hz),
  - `carrier_freq` (Hz) — если используем passband,
  - `seed`/`rng` — для стохастических блоков,
  - другие глобальные для тракта параметры.

Допускается хранить часть параметров в самих блоках (как атрибуты), но блок обязан быть самодостаточен по документации: что ожидает и что выдаёт.

### 3) Нормировки и единицы

Чтобы блоки “канал/шум/усиление” были сопоставимы, вводим минимальные соглашения:

- Если блок добавляет шум или считает SNR, он **явно** указывает:
  - по какой норме измеряется мощность (например средняя по семплам: `mean(|x|^2)`),
  - в каких единицах задаётся параметр (например `snr_db`).
- Если блок меняет уровень (attenuation/amplifier), он должен трактовать коэффициенты как безразмерные (амплитудные) и быть согласованным с выбранной нормой мощности.

### 4) Документирование контрактов блоков

Каждый блок обязан иметь краткое описание:

- входной формат (например: “bits: np.ndarray uint8 0/1”);
- выходной формат;
- какие ключи в `ctx` он читает/пишет (если использует контекст).

Это можно реализовать как docstring/README, но требование фиксируется на уровне архитектуры.

## Альтернативы

### A) Везде передавать “просто np.ndarray без соглашений”

Минусы:

- быстро появляется несовместимость блоков;
- невозможно однозначно интерпретировать графики/метрики.

### B) Ввести много классов-обёрток (`Bits`, `Symbols`, `Signal` и т.д.)

Минусы:

- перегруз для учебного проекта;
- усложнение API в ноутбуке.

## Последствия

- Блоки остаются простыми (вход/выход — базовые типы и NumPy), но совместимость повышается.
- Мета-информация сохраняется через `ctx`, не требуя множества классов.
- Легче добавлять новые модуляции/кодеры/каналы, не переписывая остальной тракт.
