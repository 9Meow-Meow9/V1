# ADR-0001: Notebook-first блоки и пайплайн (batch), минимальный базовый класс

Дата: 2026-04-21

## Контекст

Нужен учебный лабораторный стенд на Python, демонстрирующий полный тракт связи (сообщение → преобразования → канал → обратно) с возможностью:

- быстро в Jupyter/ноутбуке собрать тракт из блоков;
- параметризовать блоки через конструктор/атрибуты;
- легко смотреть промежуточные результаты и строить графики;
- избегать «макаронного» кода.

Ограничения/цели:

- **Нет задачи потоковой (streaming/real-time) обработки.** Работаем с конечными массивами/последовательностями (batch processing).
- Хотим **минимум базовых классов** (в идеале один).
- Архитектура вдохновляется GNU Radio (цепочка блоков), но намного проще: “прогнать данные по списку блоков”.

## Решение

### 1) Один базовый класс `Block`

Вводим один базовый класс `Block`, задающий минимальный контракт:

- Блок **параметризуется** через `__init__(...)` (никакой скрытой глобальной конфигурации).
- Блок выполняется как callable: `__call__(x, *, ctx=None) -> y`.
  - `x` — вход блока;
  - `y` — выход блока;
  - `ctx` — опциональный контекст выполнения (для метаданных, RNG, накопления результатов и т.п.).
- Для удобства ноутбука блок хранит последнюю обработку:
  - `self.input` (опционально, можно отключать/не использовать);
  - `self.output` — **последний выход**, чтобы в ноутбуке можно было сразу рисовать: `plt.plot(block.output)`.

> `output` считается *UX-фичей* для интерактивного использования. В коде, где важно управление памятью/сериями прогонов, можно не полагаться на сохранённые поля или отключать capture.

Рекомендуемая структура базового класса (на уровне контракта):

- `name: str` (по умолчанию от класса или задаётся пользователем);
- `__call__` вызывает внутренний `process(...)` (если захотим разделить публичный/внутренний интерфейс);
- `output` записывается после каждого вызова.

### 2) Пайплайн как простая композиция блоков

Пайплайн — это **упорядоченный список блоков**.

Предоставляем минимальный helper:

- `run(blocks, x, *, ctx=None, capture=None) -> y`.

Где:

- `blocks: Sequence[Block]` — тракт;
- `x` — вход первого блока (или `None`, если первый блок — источник);
- `ctx` — общий контекст на весь прогон;
- `capture` — *опциональный* механизм сохранения промежуточных результатов.

### 3) Наблюдаемость (observability): гибридный подход

Чтобы поддержать и простоту ноутбука, и аккуратные прогоны:

1) **Быстрый интерактивный способ:** `block.output`.
2) **Опциональный сбор результатов:** `capture` (словарь/контейнер), в который `run(...)` складывает выход каждого блока.

Рекомендуемый формат `capture`:

- `dict[str, Any]`, где ключ — имя блока (`block.name`) или композиция вида `"{block.name}.output"`.

Это позволяет:

- получать все промежуточные значения даже если блоки переиспользуются;
- контролировать, что именно сохранять;
- уменьшить зависимость от stateful-полей блоков при сериях прогонов.

### 4) Контекст `ctx`

`ctx` — опциональный объект (например dict-подобный), передаваемый во все блоки.

Назначение:

- хранить общие метаданные (например `sample_rate`, `carrier_freq`, `seed`, ссылки на RNG);
- накапливать статистику/метрики;
- выступать “каналом” для параметров, которые естественно общие для многих блоков.

При этом базовый контракт остаётся простым: блоки не обязаны использовать `ctx`.

## Альтернативы

### A) Только `.output` у блока (без capture)

Плюсы:

- максимальная простота.

Минусы:

- сложнее аккуратно проводить серии прогонов (память/перезаписи);
- сложнее сохранять результаты нескольких запусков/сценариев.

### B) Только stateless-блоки + `capture` (без `.output`)

Плюсы:

- лучше тестируемость/чистота;
- меньше скрытого состояния.

Минусы:

- хуже UX в ноутбуке: “хочу быстро посмотреть что вышло из блока” превращается в работу с отдельным объектом.

## Последствия

- Простая сборка тракта в Jupyter: пользователю достаточно импортировать блоки, создать их и вызвать `run(...)`.
- Для каждого блока есть единый понятный контракт; легко добавлять новые блоки.
- Наблюдаемость поддерживается двумя способами: быстрый (через `.output`) и аккуратный (через `capture`).
- Появляется минимальная инфраструктура (`Block`, `run`, опционально `ctx`).

## Пример целевого использования (эскиз)

```python
from modem.pipeline import run
from modem.blocks import (
    MessageSource,
    BytesToBits,
    Coder,
    Modulator,
    Channel,
    Demodulator,
    Decoder,
    BitsToBytes,
    MessageSink,
)

capture = {}

blocks = [
    MessageSource("hello"),
    BytesToBits(),
    Coder(kind="none"),
    Modulator(kind="psk", m=2),
    Channel(kind="awgn", snr_db=10),
    Demodulator(kind="psk", m=2),
    Decoder(kind="none"),
    BitsToBytes(),
    MessageSink(),
]

result = run(blocks, x=None, capture=capture)

# Быстрый просмотр:
mod = blocks[3]
mod.output  # numpy-массив/последовательность

# Или через capture:
import matplotlib.pyplot as plt
plt.plot(capture[mod.name][:200])
```

Примечание: конкретные классы/названия — предмет реализации, но паттерн использования фиксируется этим ADR.
